# 当前系统架构与技术实现说明

## 1. Scope

This document describes the current implementation state of VigilAI after the agent-platform refactor and the product-selection expansion.

The current system is no longer only an `Activity` workbench. It is a shared agent platform with multiple domain adapters.

## 2. Layered Architecture

### Experience Layer

- `/agent`
  - unified agent entry
  - shared session shell
  - domain switch between `opportunity` and `product_selection`
- legacy structured pages
  - `/workspace`
  - `/activities`
  - `/tracking`
  - `/digests`
  - `/sources`
  - `/analysis/*`
- product-selection workbench
  - `/selection/workspace`
  - `/selection/opportunities`
  - `/selection/opportunities/:id`
  - `/selection/compare`
  - `/selection/tracking`

### Shared Agent Platform Layer

Core code lives in `app/backend/agent_platform/`.

Primary responsibilities:

- session lifecycle
- turn persistence
- artifact persistence
- shared job model
- domain-aware tool routing
- conversation reply assembly

Core modules:

- `models.py`
- `repository.py`
- `artifact_service.py`
- `conversation_engine.py`
- `tool_router.py`
- `state_machine.py`

### Domain Modules

#### Opportunity domain

Code lives in `app/backend/opportunity_domain/`.

Responsibilities:

- search existing opportunity pool
- explain a selected opportunity
- propose next action from analysis + tracking state

Key tools:

- `opportunity_search`
- `opportunity_explain`
- `opportunity_next_action`

#### Product-selection domain

Code lives in `app/backend/product_selection/`.

Responsibilities:

- create research jobs
- score Taobao / Xianyu product opportunities
- persist opportunities and signals
- compare platform candidates
- track shortlisted items

Key tools:

- `selection_query`
- `selection_compare`

### Data And Adapter Layer

- `data_manager.py`
  - stable facade over split legacy opportunity storage components
  - app bootstrap dependency
- `data_manager_components/`
  - `activity.py`
  - `analysis.py`
  - `workspace.py`
- `product_selection/repository.py`
  - isolated SQLite repository for selection data
- `product_selection/adapters/live.py`
  - shared live-fetch and extraction boundary for marketplace listing pages
- `product_selection/adapters/taobao.py`
- `product_selection/adapters/xianyu.py`
  - platform adapters that prefer live extraction and downgrade to fallback when evidence quality is weak
- `analysis/*`
  - retained analysis infrastructure for the opportunity domain

## 3. Persistence Model

### Legacy tables remain active

These tables still back the opportunity domain and existing pages:

- `activities`
- `tracking_items`
- `digest_*`
- `analysis_*`

### Shared agent tables

These were added for the shared platform:

- `agent_sessions`
- `agent_turns`

## 5. End-To-End Execution Flow

### Opportunity flow

1. The user opens `/agent` or a legacy opportunity page.
2. If the user works in `/agent`, the frontend creates an `opportunity` session.
3. The backend routes the turn to opportunity tools.
4. The conversation engine builds assistant text plus artifacts such as:
   - checklist
   - shortlist
   - explanation
   - next action
5. Structured pages continue to read from legacy opportunity APIs.

### Product-selection flow

1. The user opens `/agent` and switches to `product_selection`, or opens `/selection/*`.
2. The frontend creates a `product_selection` session or starts a research job directly.
3. The backend routes the turn to:
   - `selection_query`
   - `selection_compare`
4. The service creates a research job, scores opportunities, and persists results.
5. The conversation engine returns:
   - checklist artifact
   - shortlist artifact
   - comparison artifact when applicable
6. Artifact links can jump into `/selection/opportunities` or `/selection/compare`.
7. Opportunity and tracking views surface provenance fields so the user can distinguish live extraction from explicit fallback or failed-capture states.

## 6. Migration Rules

- Old routes remain supported.
- Old tables remain readable.
- New shared-platform tables are additive.
- New product-selection tables are additive.
- `product_selection` is isolated from `Activity`.
- Shared agent sessions are the new top-level interaction model, but they do not force a rewrite of legacy structured pages.

## 7. Current Technical Tradeoffs

- SQLite remains the persistence layer for both old and new domains.
- The frontend still mixes legacy page flows and the new shared agent flow.
- The product-selection adapters now attempt live marketplace extraction and fail closed on shell-heavy or weak pages; this is not yet equivalent to a fully stable production data source.
- `DataManager` has been split internally, but the facade still carries broad legacy responsibility and is not yet a fully separate set of services.
