# VigilAI V2 闭环改造方案

## 1. 文档目标

本文用于把 VigilAI 在 `广义技术人搞钱机会聚合` 方向下的 V2，从“信息工作台”继续推进为“机会执行工作台”。

本方案只解决一个核心问题：

> 用户看到机会之后，如何持续推进，而不是收藏后遗忘。

---

## 2. 当前问题判断

当前 V2 已经具备以下骨架：

- 工作台：能告诉用户今天有哪些重点
- 机会池：能做筛选、排序、批量操作
- 详情页：能帮助用户理解机会并做初步判断
- 跟进页：能承接收藏、状态和提醒
- 日报：能把分散信息整理成摘要

但主链路仍然偏松，主要问题有 4 个：

1. 工作台更像“信息总览”，不够像“今日行动台”
2. 加入跟进仍然偏记录动作，不是推进动作
3. 跟进状态是系统态，不是业务态
4. 跟进后缺少强提醒和积压清理机制

---

## 3. V2 目标闭环

V2 后续应围绕下面这条链路推进：

```text
工作台发现重点
-> 详情页完成判断
-> 加入跟进并补全下一步动作
-> 系统在关键节点提醒
-> 跟进页处理积压与临期事项
-> 日报沉淀今日重点与待办
```

如果这条链路不能跑通，产品仍然会停留在“高质量列表站”。

---

## 4. 核心改造原则

### 4.1 先做行动闭环，再做更多内容能力

优先级必须是：

1. 看什么
2. 值不值得
3. 下一步做什么
4. 什么时候回来继续做

### 4.2 用户看到的是业务状态，不是系统状态

前台不应让用户感知：

- saved
- tracking
- done
- archived

前台只应展示真实推进阶段。

### 4.3 每次“加入跟进”都必须产出下一步动作

否则加入跟进只是延迟决策，不是进入推进。

### 4.4 工作台优先输出行动，不优先输出数据

首页首屏必须优先回答：

- 今天先做什么
- 为什么先做这个

---

## 5. 跟进状态机方案

## 5.1 前台统一状态

建议前台统一为 5 个业务状态：

- `待判断`
- `已关注`
- `准备参与`
- `已提交`
- `已放弃`

## 5.2 状态定义

### 待判断

定义：

- 先收进来，但尚未决定是否投入

典型场景：

- 从工作台收藏
- 从机会池批量加入
- 详情页看过但还未做决策

### 已关注

定义：

- 用户确认值得继续跟踪，但还未开始实际动作

典型场景：

- 认为机会值得看
- 等待规则变化
- 等待更合适的时间进入

### 准备参与

定义：

- 用户已经开始准备

典型场景：

- 准备报名
- 准备材料
- 研究规则
- 拉人组队

### 已提交

定义：

- 核心动作已完成

典型场景：

- 已报名
- 已提交作品
- 已完成任务

### 已放弃

定义：

- 明确不再推进

典型场景：

- 门槛太高
- 时间不够
- 奖励不值当
- 已被替代机会覆盖

## 5.3 状态流转

建议主流转路径：

```text
待判断 -> 已关注 -> 准备参与 -> 已提交
待判断 -> 已放弃
已关注 -> 已放弃
准备参与 -> 已放弃
```

## 5.4 系统态与前台态分离

后端如需保留：

- archived
- hidden
- deleted

可以继续保留，但不作为主前台状态展示。

---

## 6. 页面级改造重点

## 6.1 工作台

### 改造目标

从“重点概览页”升级成“今日行动台”。

### 必做改造

#### 模块 1：新增“今日先做什么”

首屏新增一个行动模块，固定展示 3 条最需要优先处理的事项。

每条包含：

- 机会名
- 原因
- 紧迫度
- 动作按钮

动作按钮建议：

- 查看详情
- 加入推进
- 稍后处理

#### 模块 2：概览卡片支持行动跳转

卡片不只展示数字，要直接跳到对应处理视图：

- 今日新增 -> 最新机会
- 高价值机会 -> 高价值视图
- 72 小时内截止 -> 紧急处理视图
- 待判断积压 -> 跟进页待判断区
- 来源异常 -> 受影响主题视图

#### 模块 3：增加积压提醒

建议在首页显式提示：

- 待判断积压数
- 已关注但无下一步动作数
- 准备参与但停滞数

## 6.2 机会详情页

### 改造目标

从“信息解释页”升级成“决策发起页”。

### 必做改造

#### 模块 1：新增决策结论卡

固定输出：

- 结论：值得做 / 观察 / 放弃
- 主要原因
- 风险点
- 推荐下一步

#### 模块 2：加入跟进改成轻表单

用户点击“加入跟进”时，弹出轻表单，至少要求填写：

- 当前阶段
- 下一步动作

可选填写：

- 提醒时间
- 备注

#### 模块 3：门槛与成本区

显式告诉用户：

- 参与门槛
- 预估投入
- 材料要求
- 个人/团队要求

## 6.3 跟进页

### 改造目标

从“我的跟进列表”升级成“我的机会推进面板”。

### 必做改造

#### 模块 1：按业务阶段组织列表

使用：

- 待判断
- 已关注
- 准备参与
- 已提交
- 已放弃

替代当前系统态分组。

#### 模块 2：新增“积压处理区”

单独展示：

- 超过 3 天未判断
- 已关注但没有下一步动作
- 准备参与但 2 天无更新

#### 模块 3：卡片强化推进字段

每张卡片优先展示：

- 当前阶段
- 下一步动作
- 提醒时间
- 距截止时间
- 最近变化

## 6.4 日报

### 改造目标

从“摘要内容”升级成“行动交付物”。

### 必做改造

日报结构建议固定为：

1. 今日总览
2. 今日最值得处理的 3 条机会
3. 今日待处理跟进
4. 今日关键变化
5. 今日风险提示

## 7. 数据字段改造建议

## 7.1 跟进实体建议新增

- `stage`
- `next_action`
- `next_action_due_at`
- `remind_at`
- `last_user_action_at`
- `abandon_reason`
- `is_digest_candidate`

## 7.2 机会实体建议新增

- `decision_hint`
- `recommended_next_action`
- `fit_summary`
- `risk_summary`
- `barrier_summary`
- `source_confidence_reason`

## 7.3 工作台聚合接口建议新增

- `today_action_items`
- `backlog_count`
- `stale_tracking_count`
- `confidence_warning`
- `affected_topics`

---

## 8. 最小上线范围

当前最值得优先推进的闭环增强只有 4 项：

### P0-1 跟进状态重构

把前台状态统一改成业务阶段。

### P0-2 详情页加入跟进弹层

加入跟进时必须补全下一步动作。

### P0-3 工作台“今日先做什么”

首屏新增行动模块。

### P0-4 跟进页阶段分组 + 积压区

让跟进页真正承担推进任务。

---

## 9. 验收标准

闭环改造完成后，至少满足：

1. 用户进入工作台后，能明确看到今天先做什么
2. 用户从详情页加入跟进时，必须形成下一步动作
3. 用户在跟进页里能按真实阶段管理机会
4. 用户能看到积压和停滞事项
5. 日报能输出行动项，而不是只输出摘要

---

## 10. 一句话总结

V2 下一步最重要的，不是继续扩页面，而是把已有页面之间的动作链做硬：

> 从“发现机会”推进到“持续执行机会”。
