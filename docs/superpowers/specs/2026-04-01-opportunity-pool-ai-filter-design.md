# VigilAI 机会池固定筛选 + AI 自然语言精筛设计

## 1. 文档目标

本设计用于为机会池补齐两类能力：

1. 面向高频真实需求的固定针对性筛选
2. 基于自然语言输入的 AI 精筛，只保留符合条件的机会

本轮只做规格设计，不直接进入实现。

## 2. 背景与问题

当前机会池已经具备列表、搜索、分类、排序、分析状态等基础能力，但用户在真实使用中仍有三个明显问题：

1. 机会数量过多，前端缺少足够“针对性”的筛选条件
2. 现有筛选更偏底层字段，不够贴近“我现在想找什么机会”的真实表达
3. 用户希望直接输入一句自然语言，由 LLM 在当前机会中做进一步筛选，而且结果必须是“筛掉不符合的，只保留符合的”

这意味着机会池不能只做成一个大列表，也不能只靠 AI 排序，而是要采用：

- 固定筛选负责高频、稳定、可解释的条件
- AI 精筛负责理解模糊目标和复合意图

## 4. 设计目标

本轮设计目标如下：

1. 保留分类，不做“无分类大列表”
2. 在机会池顶部提供一组固定、稳定、可解释的筛选条件
3. 新增 AI 自然语言精筛输入框，支持一句话描述目标机会
4. AI 精筛结果必须只保留符合条件的机会，而不是仅做排序或推荐
5. 固定筛选与 AI 精筛必须可叠加使用
6. 所有前端用户可见文案统一使用中文

## 5. 设计原则

### 5.1 固定筛选优先，AI 精筛增强

固定筛选解决的是“高频、明确、重复出现”的需求，例如：

- 我只想看黑客松
- 我只看一周内截止的
- 我只看奖金明确的
- 我只看适合独立开发者的

AI 精筛解决的是“模糊、组合、口语化”的需求，例如：

- 帮我找适合个人开发者、两周内能交付、奖励比较明确的机会
- 只保留适合快速出结果、最好线上可参加的项目
- 我想找更适合做 AI 产品 demo 的机会

### 5.2 AI 只在候选集上精筛

AI 不直接在全量库上无限制搜索，而是基于当前机会池候选集进行筛选：

1. 先使用固定筛选缩小范围
2. 再调用 LLM 对候选机会逐条判断
3. 返回仅保留的结果

### 5.3 结果必须可解释

AI 精筛后的每条保留结果，都要给出简短中文理由，至少说明：

- 为什么被保留
- 匹配了哪些条件
- 是否存在不确定项

### 5.4 前端展示全部中文

本轮以及后续新增功能，前端用户可见内容统一采用中文，包括：

- 标签
- 按钮
- 空状态
- 错误提示
- AI 解释

## 6. 固定筛选设计

### 6.1 第一层：核心高频筛选

以下筛选作为机会池默认显性筛选，直接展示在顶部区域：

1. 分类
2. 截止时间
3. 奖金区间
4. 独立开发者友好
5. 奖励是否明确
6. 投入成本
7. 来源可信度
8. 线上/远程

### 6.2 各筛选维度定义

#### 分类

来源：`category`

建议选项：

- 全部分类
- 黑客松
- 数据竞赛
- 编程竞赛
- 赏金
- 资助
- 空投
- 开发者活动

#### 截止时间

来源：`deadline_level`

建议选项：

- 全部时效
- 紧急截止
- 即将截止
- 近期开放
- 较晚截止

#### 奖金区间

来源：`prize.amount`

建议选项：

- 全部奖金
- 未知
- 0 - 500
- 500 - 2000
- 2000 - 10000
- 10000 以上

说明：

- 固定筛选应支持“奖金未知”单独筛掉或单独查看
- 后端需要将 `prize.amount` 变成可过滤条件，而不只是可展示字段

#### 独立开发者友好

来源：`analysis_fields.solo_friendliness`

建议选项：

- 全部
- 适合单人
- 需要团队
- 暂不明确

#### 奖励是否明确

来源：`analysis_fields.reward_clarity`

建议选项：

- 全部
- 奖励明确
- 奖励一般
- 奖励不清晰

#### 投入成本

来源：`analysis_fields.effort_level`

建议选项：

- 全部
- 低投入
- 中投入
- 高投入

#### 来源可信度

来源：

- 优先使用 `trust_level`
- 与 `analysis_fields.source_trust` 保持语义一致

建议选项：

- 全部
- 高可信
- 中可信
- 低可信

#### 线上/远程

来源：`location`

建议选项：

- 全部
- 线上 / Remote
- 线下 / 本地
- 混合 / 未明确

说明：

- 第一版可通过 `location` 关键词归一化得到
- 若地点字段缺失，则归入“未明确”

### 6.3 第二层：保留但不抢占主视觉的筛选

以下维度保留为补充筛选，不作为默认最高优先级展示：

- 来源
- 分析状态
- 是否已追踪
- 是否已收藏
- 关键词搜索
- 排序

这样可以避免顶部信息过载，同时保留老能力。

## 7. AI 自然语言精筛设计

### 7.1 目标

用户输入一句中文需求，系统调用 LLM 在“当前候选机会集合”中进行语义筛选，最终只返回符合条件的机会。

### 7.2 交互入口

在机会池筛选区新增一块独立模块：

- 标题：`AI 精筛`
- 输入框占位文案：`例如：只保留适合独立开发者、奖金明确、两周内截止的线上机会`
- 主按钮：`开始 AI 精筛`
- 次按钮：`清除 AI 条件`

### 7.3 AI 精筛行为

当用户点击“开始 AI 精筛”后：

1. 以前端当前固定筛选条件为基础，先得到候选机会集
2. 将候选机会的结构化信息和简要文本摘要发给 LLM
3. LLM 对每条机会返回“保留/不保留”判断
4. 前端只展示被保留的机会
5. 页面顶部显示本次 AI 精筛摘要

摘要内容建议包括：

- 识别出的筛选意图
- 候选总数
- 保留数量
- 未保留数量

### 7.4 返回结果要求

AI 精筛结果需要返回：

- `matched_items`
- `matched_count`
- `discarded_count`
- `parsed_intent_summary`
- `reason_summary`
- 每条保留机会的 `ai_match_reason`
- 每条保留机会的 `ai_match_confidence`

其中 `matched_items` 才是前端最终展示列表来源。

### 7.5 候选集上限

为了避免成本失控与结果不稳定，AI 精筛不直接处理无限数量的机会。

第一版规则建议为：

- 若固定筛选后的候选数 `<= 200`，允许直接 AI 精筛
- 若候选数 `> 200`，不直接调用 LLM，前端提示用户先缩小固定筛选范围

提示文案示例：

`当前候选机会过多，请先通过分类、截止时间、奖金区间等条件缩小范围后再进行 AI 精筛。`

这样比“静默截断前 200 条”更可控，也更符合用户预期。

### 7.6 AI 输出原则

AI 不负责生成夸张结论，而要输出简洁、可信、像中文 Agent 的判断：

- 直接说明匹配原因
- 不使用空泛套话
- 对不确定项明确标记“信息不足”或“需人工确认”

示例：

- `保留：奖励信息明确，适合单人开发，且可线上参与。`
- `保留：截止时间较近，投入成本中低，与你的快速尝试需求匹配。`
- `未完全确定：地点未写清，但从描述看更偏线上提交。`

## 8. 前端方案

### 8.1 页面结构调整

`ActivitiesPage.tsx` 调整为三段结构：

1. 关键词搜索与排序
2. 固定针对性筛选区
3. AI 精筛区

列表结果区在筛选区下方显示。

### 8.2 固定筛选区

建议将现有 `FilterBar.tsx` 升级为更面向机会判断的筛选器，至少拆出：

- 分类
- 截止时间
- 奖金区间
- 独立开发者友好
- 奖励明确性
- 投入成本
- 来源可信度
- 线上/远程

要求：

- 每个条件都能一眼看懂
- 不再只围绕“来源”展开主要视觉空间
- 清空筛选时能一次恢复默认状态

### 8.3 AI 精筛区

建议新增独立组件，例如：

- `OpportunityAiFilterPanel.tsx`

核心状态包括：

- `aiQuery`
- `aiFilterLoading`
- `aiFilterApplied`
- `aiFilterSummary`
- `aiFilteredItems`
- `aiFilterError`

### 8.4 列表展示逻辑

展示优先级：

1. 如果 AI 精筛已成功应用，则展示 `aiFilteredItems`
2. 如果 AI 精筛未启用，则展示普通列表结果

页面需明确告诉用户当前处于哪种状态，例如：

- `当前为固定筛选结果`
- `当前为 AI 精筛结果，仅保留符合条件的机会`

### 8.5 用户可见中文要求

本功能新增的所有前端文案必须直接写成中文，不新增英文默认文案，不把英文暴露给用户后再靠翻译兜底。

## 9. 后端方案

### 9.1 新增接口

新增独立接口，而不是把 LLM 精筛硬塞进现有 `/api/activities`：

- `POST /api/activities/ai-filter`

原因：

- AI 精筛是一次明确动作，不是普通列表查询
- 需要传自然语言请求体，不适合继续堆在 query string
- 需要返回结构化解释摘要，和普通列表响应形态不同

### 9.2 请求结构

建议请求体：

```json
{
  "base_filters": {
    "category": "hackathon",
    "deadline_level": "soon",
    "trust_level": "high"
  },
  "query": "只保留适合独立开发者、奖金明确、可线上参加的机会"
}
```

### 9.3 响应结构

建议响应体：

```json
{
  "query": "只保留适合独立开发者、奖金明确、可线上参加的机会",
  "parsed_intent_summary": "筛选适合单人开发、奖励明确、支持线上参与的机会",
  "candidate_count": 38,
  "matched_count": 9,
  "discarded_count": 29,
  "items": [
    {
      "id": "activity-1",
      "title": "Example",
      "ai_match_reason": "奖励明确，适合单人开发，且支持远程参与",
      "ai_match_confidence": "high"
    }
  ]
}
```

### 9.4 后端处理流程

建议新增一个 AI 精筛服务层，例如：

- `app/backend/analysis/opportunity_ai_filter.py`

流程如下：

1. 接收 `base_filters`
2. 调用现有 `data_manager.get_activities()` 获取候选集
3. 检查候选数是否超过阈值
4. 为每个候选机会构建紧凑上下文
5. 调用 LLM 执行筛选
6. 解析结构化响应
7. 返回仅保留的机会列表

### 9.5 候选机会传给 LLM 的内容

每条候选机会建议提供以下字段：

- 标题
- 分类
- 摘要
- 标签
- 奖金
- 截止时间
- 地点
- 主办方
- 来源名称
- 来源可信度
- 独立开发者友好
- 奖励明确性
- 投入成本
- 回报效率

原则：

- 优先传结构化字段和清洗后的摘要
- 不直接把冗长原始全文全部塞给模型
- 让模型做判断，而不是做重复的信息抽取

### 9.6 输出格式约束

LLM 必须返回严格 JSON，至少包含：

- `keep`
- `reason`
- `confidence`
- `uncertainties`

如果 JSON 解析失败，则视为本次 AI 精筛失败，不更新前端结果集。

## 10. 数据与字段扩展

### 10.1 `ActivityFilters` 扩展

前后端过滤对象需要新增如下字段：

- `prize_range`
- `solo_friendliness`
- `reward_clarity`
- `effort_level`
- `remote_mode`

必要时可补充：

- `organizer_keyword`
- `tag_keyword`

### 10.2 `DataManager.get_activities()` 扩展

`get_activities()` 需要补齐对应过滤逻辑：

- 奖金区间过滤
- `analysis_fields` 内字段过滤
- 地点归一化后的远程过滤

其中 `analysis_fields` 当前存为 JSON 文本，第一版可采用：

1. Python 侧取出后过滤
2. 或通过 SQLite JSON 能力过滤

第一版推荐优先选择更稳妥、改动更小的实现方式，只要结果正确即可。

### 10.3 地点归一化

为了支持线上/远程筛选，后端需要增加一个轻量归一化规则，将 `location` 映射为：

- `remote`
- `offline`
- `hybrid`
- `unknown`

这可以作为衍生字段存在内存逻辑中，第一版不强制要求数据库新增专门列。

## 11. 错误处理与兜底

### 11.1 AI 服务不可用

如果 LLM 未配置、超时、返回格式错误或调用失败：

- 不改变当前固定筛选结果
- 前端显示中文错误提示
- 用户可以继续使用固定筛选

示例文案：

`AI 精筛暂时不可用，请稍后重试，或先使用固定筛选缩小范围。`

### 11.2 无匹配结果

如果 AI 判断后没有符合条件的机会：

- 列表显示空状态
- 明确告诉用户是“AI 精筛后无符合项”
- 提供一键清除 AI 条件

示例文案：

`在当前候选机会中，没有找到符合这条 AI 条件的结果。你可以放宽要求后再试。`

### 11.3 候选过多

如果候选机会过多：

- 不触发 AI 调用
- 明确提示先缩小固定筛选范围

## 12. 测试策略

### 12.1 后端测试

新增或补充测试覆盖：

1. 固定筛选字段能正确过滤结果
2. 奖金区间过滤逻辑正确
3. `analysis_fields` 相关过滤正确
4. 地点归一化后的远程筛选正确
5. AI 精筛接口在模型返回成功时只保留匹配结果
6. AI 精筛接口在模型失败时正确返回错误
7. 候选数超限时正确拦截

### 12.2 前端测试

新增或补充测试覆盖：

1. 固定筛选项展示为中文
2. 固定筛选变更后能刷新列表
3. AI 精筛输入框、按钮、清除逻辑正常
4. AI 精筛成功后页面只展示匹配结果
5. AI 精筛失败后保留原列表并展示错误提示
6. AI 精筛空结果时显示正确空状态

## 13. 分阶段实现建议

### Phase 1

先补齐固定筛选链路：

- 前端筛选器结构升级
- 类型补齐
- 后端过滤逻辑补齐

### Phase 2

再接入 AI 精筛最小闭环：

- 新接口
- LLM 调用
- 结构化输出
- 前端应用结果

### Phase 3

最后做体验完善：

- AI 精筛摘要
- 更好的错误提示
- 更稳定的解释文案

## 14. 本轮结论

本轮确认的方案为：

- 机会池继续保留分类与固定筛选
- 固定筛选负责高频、稳定、明确的条件
- 新增 AI 自然语言精筛作为增强层
- AI 精筛的结果不是排序，而是只保留符合条件的机会
- 前端新增功能全部使用中文

本设计确认后，下一步进入实现计划编写，再开始代码落地。
