# VigilAI V2 开发实施路线图

## 1. 文档目标

本文用于把前面的 V2 产品文档收敛成一份可执行的开发实施路线图。

这份路线图重点回答：

1. V2 按什么顺序落地
2. 每一阶段要改哪些前后端模块
3. 哪些能力必须先打底
4. 如何控制范围，避免实现失焦

## 3. 实施总原则

### 3.1 先改形态，再补智能

V2 先做的是：

- 首页工作台
- 机会池
- 详情页判断能力
- 跟进闭环
- 日报输出

不是先做：

- 复杂 AI
- 花哨图表
- 大量新来源

### 3.2 先做聚合接口，再做页面拼装

V2 首页和日报都不适合继续靠前端多个接口拼装。  
优先建议在后端增加聚合型接口，减少前端页面复杂度。

### 3.3 先保证单用户闭环，再扩多人协作

V2 当前阶段可以默认是单用户、单工作空间思路。  
团队协作和复杂权限不进入第一阶段。

## 4. V2 实施阶段

## 4.1 阶段一：基础改造层

### 目标

为 V2 页面和工作流做底层数据准备。

### 后端任务

#### 任务 1：补强 `Activity` 模型

建议新增或统一以下字段：

- `score`
- `score_reason`
- `summary`
- `deadline_level`
- `trust_level`
- `updated_fields`

落点：

- `app/backend/models.py`
- `app/backend/data_manager.py`

#### 任务 2：补强详情字段

确保详情接口稳定返回：

- `full_content`
- 结构化时间线
- 推荐理由
- 相关机会候选

落点：

- `app/backend/api.py`
- `app/backend/data_manager.py`

#### 任务 3：新增跟进数据表

建议增加单用户跟进表：

- `activity_id`
- `status`
- `notes`
- `remind_at`
- `created_at`
- `updated_at`

落点：

- `app/backend/data_manager.py`
- `app/backend/models.py`

#### 任务 4：新增首页聚合接口

建议增加：

- `GET /api/workspace`

返回：

- 今日概览
- 重点机会
- 日报预览
- 趋势
- 异常来源
- 行动引导

落点：

- `app/backend/api.py`
- `app/backend/data_manager.py`

### 前端任务

#### 任务 1：扩展类型系统

更新前端类型，补齐：

- 评分字段
- 日报字段
- 工作台字段
- 跟进字段
- 来源健康字段

落点：

- `app/frontend/src/types/index.ts`

#### 任务 2：扩展 API 服务层

新增方法：

- `getWorkspace()`
- `getTrackingItems()`
- `updateTrackingStatus()`
- `getDigests()`

落点：

- `app/frontend/src/services/api.ts`

### 阶段产出

- 前后端具备 V2 所需的数据接口基础
- 不改视觉，也能开始承接 V2 页面开发

## 4.2 阶段二：工作台首页

### 前端任务

建议新增：

- `app/frontend/src/pages/WorkspacePage.tsx`

当前 `DashboardPage` 不建议硬改成工作台，建议新建页面后逐步替换默认首页。

#### 任务 2：新增工作台组件

建议新增组件：

- `WorkspaceHeader`
- `OverviewMetricsGrid`
- `MetricCard`
- `PriorityOpportunityPanel`
- `PriorityOpportunityItem`
- `DailyDigestPanel`
- `TrendChartPanel`
- `AlertSourcePanel`
- `FirstActionPanel`

建议目录：

- `app/frontend/src/components/workspace/`

#### 任务 3：新增工作台 Hook

建议新增：

- `useWorkspace.ts`

- 拉取首页聚合数据

- `/` -> `WorkspacePage`

落点：

- `app/frontend/src/App.tsx`

## 7. 开发风险与控制

### 风险 1：前端页面先行，后端接口不稳定

控制方式：

- 先定义聚合接口结构
- 再开发页面

### 风险 2：V2 范围膨胀

控制方式：

- 首页、机会池、详情页、跟进、日报之外的功能全部后置

### 风险 3：旧页面与新页面长期并存导致结构混乱

控制方式：

- 尽快完成路由切换
- 明确旧页面是兼容页还是废弃页

---

## 8. 一句话总结

V2 的开发顺序应该非常明确：

> 先打底层数据和接口，再完成首页工作台，再重构机会池和详情页，然后补上跟进与日报，最后再扩展情报中心与来源健康。

如果按这个顺序推进，V2 会稳步从“机会列表站”升级为“机会工作台”，而不会在中途失去重点。
